元の計画書は「動く最小限のスクリプト」を作る上では正しい方向性ですが、実運用に耐えるレベルにするには以下の観点が抜けています。これらを Phase 0 として明示的に加筆します。

| 項目 | 元計画書の問題点 | 修正方針 |

## 1. アーキテクチャ・技術スタック（改訂）

* **言語:** Python 3.x（構築済み）
* **コアライブラリ:**
  * `pyautogui` … マウス・キーボード操作、スクリーンショット取得
  * `keyboard` … グローバルホットキー監視（**Windowsでは管理者権限が必要な場合あり**）
  * `pygetwindow`（Windows/macOS一部対応） または macOSは `AppKit`/`osascript` … 対象ウィンドウのアクティブ状態を確認
  * `Pillow` … スクリーンショット比較によるページ遷移検知（差分検出）
  * `logging`（標準ライブラリ） … `print` ではなく正式なログ出力にする（ログレベル・ログファイル出力対応）
  * `json`（標準ライブラリ） … 設定ファイル読み込み
* **開発手法:** AI駆動開発（Claude Code / Codex へのプロンプト指示によるコード生成、人間によるレビュー必須）
* **配布形態（任意）:** `PyInstaller` で単一exe/appにビルドし、Python環境がないPCでも起動可能にする（Phase 5として後述）

### OS別の注意点（重要・元計画書では末尾の一言のみだった点を前倒し）

* **macOS:** `keyboard`ライブラリのグローバルフックには「アクセシビリティ」権限に加え、Sequoia以降は「入力監視（Input Monitoring）」権限も別途必要。ターミナル/iTerm自体に権限を付与しないと**無反応のまま原因が分かりにくい**ため、Phase 1で必ず動作確認する。
* **Windows:** 対象アプリが管理者権限で起動している場合、非管理者権限のスクリプトからは`pyautogui`のキー送信が届かないことがある（UAC分離）。この場合はスクリプト側も管理者権限で実行する必要がある。
* **Windows/macOS共通:** マルチモニタ環境では`pyautogui`の座標系がずれることがあるため、座標クリック方式より**キーボードショートカット方式を優先**する。

## 2. 待機時間の設計方針：固定間隔ではなくランダム幅を持たせる（重要・明記）

元計画書は`interval_seconds`という**単一の固定値**で待機時間を扱う設計でしたが、これは以下の理由から避けるべきです。

* **不自然な機械的挙動になる:** 完全に一定間隔でキーが送信されると、実際の読書ペース（文章量や図表の有無で読了時間は変動する）とかけ離れた挙動になる。
* **対象アプリ側の検知リスク:** サービスによっては「一定間隔での機械的操作」を不正利用パターンとして検知する場合があるため、固定間隔は避けたい。
* **柔軟な調整ができない:** 固定値だと「速く読みたいページ」「じっくり読みたいページ」の差を表現できない。

### 採用する設計：`min_interval` 〜 `max_interval` のランダム幅

`config.json`側で最小値・最大値のみを指定し、実行時に毎回その範囲内でランダムな待機時間を算出する。

```json
{
  "target_window_title": "対象アプリのウィンドウタイトル",
  "turn_key": "right",
  "min_interval": 8,
  "max_interval": 14,
  "jitter_distribution": "uniform",
  "max_turns": null
}
```

* `min_interval` / `max_interval`（秒）: 待機時間の下限・上限。この範囲内で毎回ランダムに待機時間を決定する（`random.uniform(min_interval, max_interval)`）。
* `jitter_distribution`: 将来的な拡張余地として、単純な一様分布（`uniform`）だけでなく、平均値付近に集中しやすい正規分布（`gaussian`）も選べるようにしておくと、より自然な読書リズムに近づけられる（初期実装では`uniform`のみで十分）。
* 毎回のループで新しい待機時間を算出し、固定値をキャッシュしないようにする点がポイント。

### AIへの指示例（Phase 2の指示に追記する形）

> 「待機時間は固定値ではなく、`config.json`の`min_interval`と`max_interval`（秒）の範囲内で、ループのたびに`random.uniform()`でランダムに算出してください。算出した待機時間はログに毎回出力し、後から実際の間隔のばらつきを確認できるようにしてください。」

この方針は Phase 2（コアロジック実装）以降のすべての待機処理に適用する前提とする。

## 3. フェーズ別実施案（改訂）

### Phase 0: 対象操作の実地検証（新規追加）

元計画書ではPhase 1でいきなり環境構築に入っていますが、**対象アプリの操作方法を先に確定させないと、後工程の手戻りが大きくなります。**

* **実施事項:**
  1. 対象アプリを手動で操作し、ページめくりが「キーボード（矢印/Space/PageDown等）」「マウスクリック」「スワイプジェスチャー」のいずれで行えるかを特定する。
  2. キー操作が可能な場合はそれを最優先で採用する（クリック座標方式は解像度依存で壊れやすいため）。
  3. ページめくり後に画面がどう変化するか（例: ページ番号表示、レイアウトの変化）を目視確認し、後述の「めくれたことの検証ロジック」の判定基準を決める。

* **成果物:** 操作方法メモ（使用キー、想定待機時間の目安、画面変化の特徴）

---

### Phase 1: 要件定義と基礎環境のセットアップ

* **実施事項:**
  1. **依存関係の導入:** `requirements.txt` を生成し、仮想環境（`venv`）内にインストール。
  2. **OS権限の事前確認（前倒し）:** `keyboard`ライブラリが実際にグローバルフックを取得できるか、最小限のテストスクリプトで検証する（例: F9押下でコンソールに出力するだけのスクリプト）。ここで権限エラーが出た場合、先にOS側の設定を済ませる。
  3. **ファイル構成の決定:** `main.py`, `config.json`, `logger_setup.py`, `window_utils.py`（フォーカス確認用）, `tests/` ディレクトリを用意。

* **AI（Claude Code/Codex）への指示例:**
  > 「Pythonでデスクトップ自動化スクリプトを作成します。`pyautogui`, `keyboard`, `pygetwindow`, `Pillow` を使うための`requirements.txt`と、`main.py` / `config.json` / `logger_setup.py` / `window_utils.py` / `tests/` を含む基本構成を作成してください。ログは標準の`logging`モジュールを使い、コンソールとファイルの両方に出力する設定にしてください。」

---

### Phase 2: コアロジックの実装（プロトタイプ開発）

* **実施事項:**
  1. 待機時間を**固定値ではなく範囲（例: 8〜12秒のランダム）**で扱うロジックを実装（`config.json`で`min_interval` / `max_interval`を指定）。
  2. 待機完了後、Phase 0で確定したキー操作を実行する処理を実装。
  3. **フォーカス確認処理:** キーを送信する直前に対象ウィンドウがアクティブかどうかを確認し、アクティブでなければ一時停止してログに警告を出す（誤って他アプリにキー入力してしまう事故を防止）。
  4. ループ処理（無限ループ、または`max_turns`で回数上限を設定可能に）。

* **AIへの指示例:**
  > 「`main.py`に、`config.json`から読み込んだ`min_interval`〜`max_interval`秒のランダムな待機時間ごとに指定キーを送信する関数を実装してください。キー送信前に`window_utils.py`の関数で対象ウィンドウ（`config.json`の`target_window_title`）がアクティブかどうかを確認し、非アクティブなら送信をスキップして警告ログを出す処理を追加してください。」

---

### Phase 3: 制御機能・信頼性向上機能の追加

* **実施事項:**
  1. **ホットキー制御:** 「F9で開始/一時停止」「F10で完全終了」をグローバルホットキーで実装。
  2. **動的パラメーター読み込み:** 待機時間・対象キー・対象ウィンドウ名を`config.json`から読み込み（ハードコード禁止）。
  3. **ページ遷移検証（新規・信頼性向上のための重要機能）:** キー送信の直前と直後でスクリーンショットを取得し、差分（ピクセル比較 or ハッシュ比較）が一定値以下であれば「ページがめくれなかった」と判定し、リトライまたは警告ログを出す。これにより、対象アプリがフリーズ/フォーカス外れ等で無反応になっていることを検知できる。
  4. **異常時の自動停止:** 一定回数連続でページ遷移が検出できない場合、無限にキーを送り続けるのではなく自動的に一時停止し、ユーザーに通知する。

* **AIへの指示例:**
  > 「現在のループ処理に、`keyboard`ライブラリでF9（開始/一時停止トグル）とF10（終了）のグローバルホットキーを追加してください。また、キー送信の前後で`pyautogui.screenshot()`を用いて対象ウィンドウ領域のスクリーンショットを取得し、`Pillow`で差分を比較して『ページが変化したかどうか』を判定する関数を実装してください。3回連続で変化が検出されない場合は自動的に一時停止し、ログに警告を出してください。」

### Phase 4: テスト・調整と実運用

* **実施事項:**
  1. **単体テスト:** 待機時間計算、設定ファイル読み込み、差分検出ロジック等、OS操作を伴わない純粋なロジック部分は`pytest`でユニットテスト化する。
  2. **フォーカス管理の実地テスト:** 対象アプリを最前面/非最前面に切り替えながら、意図通りにスキップ・警告が出るか確認。
  3. **タイミング調整:** 実際の読書速度に合わせて`min_interval`/`max_interval`をチューニング。
  4. **例外処理:** 画面解像度変更、キーボードフックの権限エラー、対象ウィンドウが閉じられた場合など、想定される異常系ごとに`try-except`で個別にハンドリングし、どのエラーかログで区別できるようにする（全て一括で`except Exception`にしない）。
  5. **長時間稼働テスト:** 数十分〜数時間の連続稼働でメモリリークやスクリーンショット取得の遅延蓄積がないか確認。

* **AIへの指示例:**
  > 「`config.json`の読み込み処理と待機時間計算ロジックに対する`pytest`のユニットテストを作成してください。またメインループには、`FileNotFoundError`（設定ファイル欠損）、`keyboard`権限エラー、対象ウィンドウ消失の3種類を個別に`except`節で捕捉し、それぞれ異なるログメッセージを出すようにリファクタリングしてください。」

## 5. 上級プログラマーへの推奨事項（加筆）

* **フェイルセーフ:** `pyautogui.FAILSAFE = True`（マウスを画面四隅に移動で緊急停止）は必ず維持する。加えて、F10による**能動的な緊急停止ホットキー**も併用することで、マウス操作ができない状況（フルスクリーンアプリ等）でも止められるようにする。
* **座標クリックよりキー操作を優先:** ウィンドウサイズ変更・DPIスケーリング変更でクリック座標がずれるため、可能な限りキーボードショートカットを使う設計にする。座標クリックが必須の場合は、起動時にウィンドウサイズを検出して座標を動的計算する。
* **「一定時間経過後にめくる」だけで良いか再検討:** 完全固定間隔だと、対象アプリ側で「一定間隔の機械的操作」を検知される可能性がある場合や、単に不自然な読書ペースになる場合があるため、上記の通りランダム幅を持たせることを推奨。
* **ログの保存:** 稼働ログをファイルに残しておくと、「途中で止まっていた」等のトラブル時に原因切り分けがしやすい。
* **利用規約の確認:** 対象アプリが電子書籍サービス等の場合、自動操作がサービス利用規約に抵触しないか事前に確認することを推奨（技術的な話とは別に、運用上のリスクとして明記）。
